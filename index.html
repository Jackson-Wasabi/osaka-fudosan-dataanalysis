ポートフォリオの目的

この分析は、中古マンションを「安く仕入れて再販する／投資する」担当者のための道具だ。
彼らの仕事は、まずどのエリア（駅）の物件を調べに行くかを決めること。
だが、大阪市には150以上の駅があり、その全部の相場を人手で当たるのは現実的ではない。
かといって勘で絞ると、「安いには訳がある」物件を掴むリスクが残る。

そこで「どの駅から調べ始めるべきか」を、誰がやっても同じ結果に辿り着けるよう、公開データだけで順位づけした。
出すのは“買うべき物件”ではなく、“次に足を運んで調べる価値がある駅”。

具体的には、大阪市の中古マンション成約データを起点に 151駅 → 10駅 まで自動でスクリーニングし、“安いには訳がある”逆選択の罠まで可視化した。
以降のページでは、その結果・設計上の判断・自己検証の順で、考え方ごと辿れるようにまとめた。

1. 成果（何を解決したか）

人手で当たれば151駅ぶんの相場調査が必要なところを、調査すべき10駅まで自動で絞り込む。

2,106件2025年の対象取引
→
151駅相場比較が可能
→
66駅10件以上で安定比較
→
本命 10駅調査対象を約1/14に圧縮
配点パターン 仕入れ調査優先 Top5（本命★）
バランス型（主スコア） 堺筋本町・玉造・大国町・松屋町・中之島
割安重視型 玉造・大国町・堺筋本町・伝法・森ノ宮
リスク重視型 堺筋本町・玉造・大国町・松屋町・谷町四丁目

配点パターンを切り替えると順位（＝仕入れ調査優先駅）が変わる。堺筋本町・玉造・大国町はどのパターンでも上位＝頑健な本命。割安重視では伝法・森ノ宮、リスク重視では谷町四丁目が浮上する（配点の意図に応じて狙いが動く）。

※ スコア最上位は平野（割安重視85.6点）だが、旧耐震×改装不明が86%＝典型的な「安いには訳がある」逆選択のため本命から自動除外。詳細は §2。

2. 設計で大事にした4つの判断

問い どう考えて、どうしたか
Q1. なぜ一番安い物件を本命にしない？ 安い物件ほど旧耐震×改装不明（訳あり）が多い＝逆選択。額面の安さで選ぶと地雷を掴むので、建物リスクを可視化して本命から外した（例：平野）。
Q2. なぜ一番"当たる"モデルを使わない？ 精度1位のFは郊外を過大予測して「偽の割安」を作り順位を壊す。私の目的は価格当てでなく駅の順位づけなので、Fは見送り、2位でも順位が安定するCを選んだ。
Q3. この結論はどこまで信じていい？ 全数値はコードから再現可能。ただし元データは「成約＝売れ終わった取引」なので、出せるのは“買い”でなく“次に調べる候補”まで。
Q4. なぜ少数データの激安駅を上位にしない？ まぐれ安値を上位に出したくないので、取引10件未満は外し、件数を信頼性スコアに加点した。

5. データ前処理 ― 外れ値・欠損をどう判断したか

ここは一番こだわった工程。ルールで一律に切るのではなく、列ごとに「なぜそう処理するか」を決めてから手を動かした（全工程の詳細は docs/eda_report.md）。

欠損データ：「埋める」か「不明のまま残す」かを、列ごとに使い分け

  • 改装の有無は95.7%が空欄（大阪市の元データの特性）。ここを無理に推測で埋めると、ありもしない情報を作ってしまう。だから埋めずに「不明」という印を付け、その駅のデータの信頼性やリスクを測る材料に使った。
  • 建築年（1.9%）・駅からの徒歩分（2.8%）はごく一部だけ欠損。こちらは周辺の標準的な値（中央値）で補い、「補完した」という印を残した。※このとき未来の取引データは使わず、過去のデータだけで埋めている（“答えを先に見てしまう”状態を防ぐため）。
  • ほかにも「徒歩30分〜60分」のような曖昧な表記（47件）や、まったく同じ行の重複（330件）も見つけ、数値に直す・重複の印を付ける、で対処した。

外れ値（極端な値）：数式で機械的に消さず、中身を見て判断

  • 統計的に「外れ値」と判定された585件も、すぐには消さなかった。まず価格のばらつきを扱いやすくする変換をかけ、そのうえで物理的にあり得ない値だけを除外。極端な1件1件は実際にデータを目で見て、「事故物件や再建築不可なのか、それとも入力ミスなのか」を見極めた。
  • そもそも分析の対象を面積20〜60㎡・徒歩20分以内・築5〜60年・価格500万円以上に絞ったので、対象外の極端な物件は自然と外れる形にした。
  • 一方で正直な弱点も残る。安すぎる物件用のフィルタは価格の下限設定と重複して意味がなかったので外したが、高すぎる側のチェックは未対応であることを隠さず明記し、次の改善課題とした。

6. モデル評価（テスト＝2025年・n=2,097）

モデル MAE（平均誤差・円/㎡） hit10（±10%以内の的中率） 採用
Baseline（駅中央値そのまま） 186,117 0.176
Model A（面積+築年+徒歩） 135,409 0.267
Model B（A+改装+新耐震） 135,611 0.269
Model C（B+駅中央値） 120,567 0.269 ✅ 採用
Model F（C+時点index） 101,093 0.329 ✗（Q2の理由で不採用）

※ MAE＝予測した㎡単価が実際の価格から平均で何円ズレたか（小さいほど高精度）。hit10＝予測が実勢価格の ±10% 以内に収まった取引の割合（大きいほど良い）。

選んだCは Baseline 比 MAE 約35%改善。駅ごとの相場（駅中央値㎡単価）は、未来の情報が混ざらないようその検証で使う"学習期間"のデータだけから計算している（過去を学んで未来を当てる設定なので、未来の答えを先取りする「リーク」を防ぐため）。

また公示地価（国が出す土地の基準価格）は、あえて予測モデルには入れず、スコア側（地価補足15点）でだけ使った。地価を予測に入れると「地価が高い駅だから価格も高い」と学習してしまい、本来見つけたい“相場より割安”というシグナルを地価で説明し尽くしてしまうためだ。→ 詳しい配点は「調査優先度スコア」にまとめている。

※ §1ファネルの 2,106件は2025年の全対象取引、評価の n=2,097 は予測特徴量が揃う行（差の9件は特徴量欠損で評価対象外）。

  • 2025年だけに頼らない（時期をずらして再検証）：学習と検証の期間の区切りをずらしてもう一度試し（2021〜23年で学習 → 2024年で答え合わせ）、的中率 hit10 が 0.27→0.32 と大きく崩れないことを確認した（たまたま2025年だけ当たった、ではない）。
  • 何で効いているかも確認：標準化係数（ML.WEIGHTS）を見ると効くのは築年（-0.32で支配的）と駅相場で、改装はほぼ0＝価格に効かないとデータで裏取りした（sql/bqml/04_inspect_weights.sql）。

7. 調査優先度スコア

調査優先度 = 割安度45 + 市場信頼性25 + 地価補足15 + データ品質15 − リスク減点20  （0〜100にクリップ）

配点に絶対の正解はないので、3パターン（バランス／割安重視／リスク重視）振っても上位に残る駅を 安定候補、その中で建物リスクが低い駅を 本命 とした（感度分析）。

▶ 配点の詳細（5軸の作り方・3パターンの重み・設計判断）：docs/scoring_methodology.md

※ 割安度の元になる「駅相対乖離」は、市場全体と駅内で二段に正規化して算出（一様な時点要因と駅ごとの空間要因を相殺し、その駅の相場に対する割安度を測る）。

8. 結果を疑う ― 上位駅の自己検証

ランキングを出して終わりにせず、自分が出した上位10駅を「逆選択の指紋」（旧耐震率・大型物件への偏り・薄商い・極端な外れ値の集中）で再検証した。結果は 本物っぽい割安が4駅／逆選択が濃厚な1駅（平野）／要・現物確認が5駅。「出した結果を自分で疑う」ところまでを一次スクリーニングの責任範囲とした（詳細：docs/verification_notes.md）。

11. 分析の限界

  • 成約データであり売出ではない。最終判断は現物・謄本・管理状態・権利関係の確認が前提。
  • 対象は中古マンション・面積20〜60㎡・徒歩20分・築5〜60年に限定（市場の約32%）。全市場ではない。
  • 改装情報は9割超が欠損（大阪市で95.7%）。「不明」をリスクとして扱った（逆選択判定の前提）。
  • 「割安＝相場より下振れ」は好機か衰退かを単独では判別できない。上位駅の逆選択検証（docs/verification_notes.md）と現物確認が前提。

今後の改善

  • 売出データ・管理状態・成約遅延の補正の取り込み
  • 駅相対の外れ値フラグ（個票レベルの異常検知）
  • 駅×時点トレンドの慎重な導入（順位の歪みを増やさない範囲で）
  • 予測モデルの特徴量拡充：間取りなど未使用の属性や、所在階・向き・総戸数の追加取得で、的中率（hit10）の底上げを図る
  • 対象セグメントの横展開：面積20〜60㎡（市場の約32%）に絞った枠組みを、60㎡超のファミリー向けなど別セグメントへ同手法で広げる
  • 定期更新の自動化：四半期ごとに公開データを取り込み直し、dbt → BigQuery ML → Tableau を自動で再実行して順位を更新するパイプライン化

制作にあたって（使用ツールと役割分担）

本プロジェクトの実装にはAIコーディング支援（Claude Code）を活用している。一方で、分析設計の中核 ―― 分析対象の絞り込み、予測特徴量とスコア項目の分離、モデルの採用判断（Model C vs F）、調査優先度スコアの配点、公開データの限界の線引き ―― はすべて自分で判断し、その理由を docs/decision_log.md に記録した。コードを書く速度をAIで上げつつ、「何を・なぜそう判断したか」を自分の言葉で説明できる状態を重視した。
