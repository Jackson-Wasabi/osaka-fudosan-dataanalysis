この手順書のゴール

分析・dbt・BQML・Tableau の構築はすでに完了しています。残る作業は「新しく作る」ことではなく、正しさを検証し、伝わる形に整え、自分の言葉で説明できる状態に仕上げることです。本書はその全工程を、順番どおり実行すれば誰でも同じ結果に辿り着くように書いています。

最初に覚える3原則

  1. 数字は必ず裏取りする ― PDF・README・Tableauに載る数値は、実テーブルか docs/ のログで確認してから書く。AIが生成した数字をそのまま使わない。
  2. 成果物は増やさず磨く ― 3枚PDF・Tableau・GitHubの3点で十分。新しい資料を増やすほど数値の矛盾リスクが増える。
  3. ゴールは「自分の言葉で説明できる状態」 ― 「Claude Codeが作ったので分かりません」は不合格。処理の目的・入力・出力・判断理由を言えればよい（コード暗記は不要）。

このプロジェクト特有の重要な事実（暗記すべき数値）

  • 学習期間 = 2021〜2024年 / 検証期間 = 2025年（成約データの提供開始が2021年。時系列で分割し、未来データで検証＝リークなし）
  • 技術スタック = BigQuery + dbt + BQML + Tableau（前処理・特徴量はSQL/dbt、検証はPythonスクリプト）
  • 絞り込み = 大阪市・面積20〜60㎡・徒歩20分・築5〜60年 → 2,106件 → 151駅 → 66駅 → 本命10駅
  • 採用モデル = Model C（MAE 約12.1万円/㎡・hit10 0.269）。Baseline(18.6万)比 約35%改善。Model F は精度1位(10.1万)だが郊外を偽の割安にするため不採用。
  • 調査優先度スコア = 割安45 + 信頼25 + 地価15 + 品質15 − リスク20（0〜100にクリップ）

1-1. 数値の一致を確認する

「Tableauに出ている数字」と「BigQuery/Pythonで再計算した数字」が一致するかを突き合わせる。1つでもズレたら原因を特定するまで先に進まない。

  • 2025年の対象取引件数（Tableau表示 2,106 と一致するか）
  • 相場比較可能な駅数（151）／10件以上の駅数（66）／本命駅数（10）
  • Model C の評価値（MAE・hit10）が docs/decision_log.md の記録と一致
  • 調査優先度スコア上位（玉造84.1・大国町82.4 など）がTableau表と一致

1-2. 情報漏洩（リーク）が無いことを確認する

面接で最も突かれる箇所。次の3点を「なぜ漏れていないか」言える状態にする。

項目確認すること
駅中央値㎡単価各Foldの訓練期間のデータだけから算出している（検証年2025の値は使っていない）。これが予測特徴量に入っても未来情報を見ていない。
公示地価予測モデルには入れていない。地価補足スコア(15点)としてのみ使用。理由を説明できること。
欠損補完（築年・徒歩分）補完に使う中央値も訓練期間のみから計算。検証データを見て埋めていない。

1-3. ロジックの向き・妥当性を確認する

  • 割安度のプラス／マイナスの向きが正しい（相場より「下」が割安＝高スコア側）
  • 取引件数が少なすぎる駅を高評価していない（10件未満は除外、件数を信頼性に加点）
  • 外れ値・欠損の処理を1つずつ説明できる（改装情報94%欠損→「不明」をリスクとして扱う 等）
  • スコア配点（45/25/15/15/−20）を電卓で1駅ぶん再現できる
  • 平野が「スコア1位なのに本命外」である理由（旧耐震86%＝逆選択）を説明できる

6-2. 必ず答えられるようにする質問（このプロジェクト特有）

質問回答の骨子
なぜ公示地価を予測モデルに入れない？地価は「割安さ」と相関しエリアの結果を先取りしてしまう。割安の判定軸を汚さないため、予測ではなく地価補足スコア(15点)に分離した。
駅中央値㎡単価を入れたらリークでは？各Foldの訓練期間のみから算出。検証年2025の値は使っていないので未来情報は見ていない。
なぜ一番当たるModel Fを使わない？Fは精度1位だが郊外を過大予測し「偽の割安」を作り順位を歪める。目的は予測精度でなく駅の順位付けなので、順位が安定するModel Cを採用。
なぜMAEも見る？（MAPEだけでない）MAPEは低価格帯で誤差が膨らみやすい。円/㎡の絶対誤差(MAE)を併用し実務的な外れ幅も確認した。
取引件数が少ない駅は？10件未満は除外し、件数を信頼性スコアに加点。少数データのまぐれ安値を抑える。
一番安い物件を本命にしない理由は？安い物件ほど旧耐震×改装不明（訳あり）が多い＝逆選択。これを可視化して自動除外（平野が好例）。
外れ値・欠損の扱いは？改装情報94%欠損→「不明」をリスク扱い。築年・徒歩分は訓練期間中央値で補完しフラグ付与。
AIに任せた部分と自分の判断は？コード生成はAI、特徴量の分離・モデル採用・スコア配点・限界の線引きは自分の判断。各判断は decision_log に記録。
実務で使うなら何を足す？売出データ・管理状態・成約遅延の補正・駅相対の外れ値フラグ。
分析の最大の弱点は？「割安＝下振れ」は好機か衰退かを単独では判別できない。だからStep12の逆選択検証と現物確認を前提にしている。
「Claude Codeが作ったので分かりません」は絶対NG。コード暗記は不要だが、各処理の目的・入力・出力・判断理由は言えること。
