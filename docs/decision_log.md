# 判断理由ログ (decision_log)

## 2026-06-12 — Step 5 EDA: 外れ値・欠損の処理方針（D-ID制・すべて提案中）

以後、staging 実装のSQLコメント・eda_report.md とこのIDで相互参照する。

- **D-001（不採用）** 大阪市スコープでは範囲表記が1件（1H30～2H）のみ、かつ明確に20分超。Step6の「徒歩20分以内」フィルターで自動除外されるため、stgでの変換処理は不要と判断。
- **D-002（採用）** 外れ値の除外は「物理的に説明できるミスのみ」に限定し、IQRフェンス自動フラグ（outlier_pps_flag）は不採用。具体的な除外対象: 面積100m²超（D-003）・㎡単価5万円未満。グレーゾーン（天神橋筋六丁目964万/m²等）は証明手段がないため残す。モデルへの影響は目的変数のlog変換（log_price_per_sqm）で抑制する（案A+B採用）。
- **D-003（採用）** 面積100m²超は scope_flag=FALSE・excluded_reason='area_out_of_scope'。100m²超の除外閾値のみ今回確定。分析に使う実際の上限（20〜55m²等）はStep 6で別途確定する。
- **D-004（不採用）** 大阪市スコープでは「戦前」表記が0件のため実装不要と判断。
- **D-005（採用）** 全21列が一致する行219件（0.9%）を全件保持し `potential_dup_flag=1` で識別するにとどめる。理由: データに部屋番号・フロア・正確な住所がなく「二重登録」と「別部屋の同時成約」を区別できない。誤って集約した場合は実在する別取引を消す不可逆なミスになるのに対し、重複を残した場合は駅別中央値・件数への影響が0.9%にとどまり軽微。誤りの方向性として「残す」の方が害が小さいと判断。
- **D-006（採用・案A）** 駅名正規化は (1)括弧除去 (2)「駅」除去 (3)mapping CSV の3段階。rawの表記をそのまま保持し、stgで変換する。mapping CSV 7行: 四天王寺前夕陽ケ丘→四天王寺前夕陽ヶ丘 / ＪＲ難波→JR難波 / なんば→難波 / あびこ→我孫子 / 鶴ケ丘→鶴ヶ丘 / ＪＲ淡路→JR淡路 / ＪＲ野江→JR野江。mapping後は駅名での不一致0件の見込み。残る station_joined=0 は駅名が空欄の469行のみ。徒歩分の信用性はmappingの影響を受けない（同一物理駅の表記変換のみ）。
- **D-007（採用）** 改装空欄23,558件（95.7%）は renovation_unknown_flag=1。改装済み=renovation_done_flag=1。「未改装」という値がデータに存在しないため空欄を未改装と断定しない。renovation_unknown_flagは予測モデルに入れず、データ品質スコア・リスク減点で使う。
- **D-008（採用・案B）** 築年数マイナスは2件（0.01%）のみ（min=-2）。竣工前売買か誤記と判断。building_age_years=0 に補正 + is_negative_age=1 フラグを付与してscope内に残す。モデルへの悪影響を防ぎつつデータ損失を最小化。is_negative_ageはモデル特徴量には入れず、リスク減点 -2点の材料とする。

- **D-009（採用・案A+B）** リーク防止として訓練期間ベースの時系列分割 + ウォークフォワード検証を併用する。Fold A（本番）: 訓練=2021-2024・テスト=2025。Fold B（検証）: 訓練=2021-2023・テスト=2024。駅別中央値・IQR等のデータ依存統計量は各Foldの訓練期間のみで算出しテスト期間に適用する。両FoldのMAEを比較し年をまたいだ安定性を確認する。面積100m²・価格500-8000万円等の業務固定閾値はリーク懸念なし。
- **D-010（採用済み・実施済み・ユーザー指摘起点）** 分布・外れ値のEDAは分析対象スコープ（大阪市）で実施する。府全体は形式チェック（範囲表記・戦前・重複等の洗い出し）のみに使う。同日中に大阪市版で全面再実行済み。**効果の実証: ㎡単価P99が府全体158万→大阪市186万円/m2**と大きく異なり、府全体閾値では大阪市の正常な高額物件を誤って外れ値扱いするところだった。

## 2026-06-12 — Step 3〜4: 投入方式の判断

### 判断1: GCPプロジェクトを新規作成（osaka-fudosan-dataanalysis）
- ユーザー指示。旧プロジェクト osaka-condo-analysis とテーブルが混ざらず、v9の成果物が独立して説明できる。課金は同一アカウントに紐付け（後のBQMLに必要）。

### 判断2: raw_transactions は大阪府全体47,386行を1テーブルに as-is 投入
- v9手順書の原則「rawは元CSVをそのまま保存」に従い、大阪市への絞り込みは staging で行う（旧作業は投入時に大阪市へ絞っていたが、絞り込み条件の変更に弱いため変更）。
- 5年分は同一スキーマのため1テーブルに追記ロード。年は trade_period 列から復元可能で情報損失なし。

### 判断3: 駅マスタ・地価は年別テーブル（_2024/_2025）で分離投入
- ユーザー指示「統合せず」に従う。年を跨ぐ結合・差分確認は staging/intermediate で行う。

### 判断4: 全列 STRING でロード
- 型解釈（例: コード"01"→数値1で先頭ゼロ消失）による意図しない値の変化を防ぐ。型変換は staging に集約し、変換ロジックを dbt で説明可能にする。

### 判断5: raw 列名は英語 snake_case に置換（位置対応）
- BigQuery の列名制約で日本語ヘッダーの「：」「（）」が使用不可のため。元の日本語カラム名との対応は data/samples/README.md のカラム一覧が対応表を兼ねる。

### 判断6: GIS は CSV（lon/lat列）方式、GEOGRAPHY型は不採用
- 最終出力が Tableau 地図のため lat/lon 数値列が最も確実。距離計算は staging で ST_GEOGPOINT を作れば可能。
- 駅の代表点は LineString 座標列の中央要素による簡易中点（数十m程度の近似）。駅単位集計・1000m圏地価の用途では影響なし。元GeoJSONは data/gis/raw に保持。

### 判断7: 成約CSVは cp932→UTF-8 の変換コピー（TEMP）でロード
- BigQuery が cp932 を読めないため。raw ファイルは無変更で保持し、変換は文字コードのみ（内容無変更）。

## 2026-06-12 — Step 2: データ取得

### 判断1: 不動産情報ライブラリはブラウザと同一の内部APIで取得した
- サイトのダウンロードボタンが呼ぶ内部API（/in-api/api-aur/aur/csv/transactionPrices）を、サイトが全ブラウザ訪問者に配布している公開キーで呼び出した。これはブラウザでダウンロードする操作のHTTP的な等価物であり、個人APIキー登録は不要。
- 取得物が旧作業の手動ダウンロードCSVとMD5一致したため、「同じ条件・同じデータ」の再現を確認できた。

### 判断2: 2020年の空CSVは削除せず保持する
- 成約価格情報は2021年第1四半期から提供のため、2020年指定では0件（ヘッダーのみ2バイト）になる。
- 「2020年分は公開データに存在しない」という調査結果自体が記録価値を持つため、空ファイルを証跡として data/raw に残し、分析からは除外する。

### 判断3: 2022年CSVのMD5不一致は問題なしと判断
- 旧新で行集合は9,107行完全一致、並び順のみ相違。サーバ側の返却順が非決定的なだけでデータ内容は同一。再取得や調査は不要。

### 判断4: 駅データはN02の2024・2025の両年を保持
- ユーザー指定（2024〜2025）どおり両年取得。駅の新設・廃止・名称変更の差分確認が可能になり、マスタ鮮度の説明材料になる。

### 判断5: 取引CSVは年単位・統合なしで保存
- ユーザー指定どおり。rawは「取得単位のまま」が原則で、統合はstaging以降のdbtで行う方が再現性・説明性が高い。
